# Omni-CSX V2 Agent Guide

## 1. 项目阶段定义

当前项目已经完成 V1 基线。V1 的定位是：
- 多平台聚合接待
- 统一订单 / 物流 / 售后上下文
- AI suggested reply + 人工确认发送

V2 的定位是：
**从“会答问题”升级为“会接待 + 会推荐 + 会跟单 + 会运营”的客服运营中台。**

## 2. V2 必做模块

V2 必须新增以下模块：
1. 商品推荐中心
   - 主推荐
   - 替代推荐
   - 连带推荐
   - 推荐话术生成
2. 智能跟单中心
   - 咨询未下单
   - 未付款
   - 发货异常
   - 售后后关怀
   - 跟单任务生成
3. 客户标签与画像中心
   - 意向标签
   - 商品偏好标签
   - 交易标签
   - 风险标签
   - 客户画像快照
4. 客户运营中心
   - 人群筛选
   - 活动任务
   - 触达模板
   - 召回任务
   - 运营任务执行
5. 基础运营分析中心
   - 会话量
   - 首响
   - AI 建议采纳率
   - 高频咨询问题
   - 高频咨询商品
   - 推荐使用率
   - 跟单任务执行率
6. 轻量风险提醒
   - 负面情绪
   - 投诉倾向
   - 黑名单标记
   - 停止营销建议

## 3. V2 工作流

### 推荐工作流
会话 -> 识别购买意图 -> 读取客户画像 -> 取候选商品 -> 生成推荐理由 -> 生成推荐话术 -> 人工确认

### 跟单工作流
会话 / 订单事件 -> 判断场景 -> 生成任务 -> 生成文案 -> 人工执行

### 运营任务工作流
筛选目标人群 -> 生成模板 -> 预览 -> 人工确认 -> 创建运营任务

## 4. 关键工程约束

1. 先做数据库 migration、ORM、schema、service、API，再做前端页面。
2. 所有新增功能都必须写审计日志。
3. 所有平台相关逻辑只能通过 provider 层访问。
4. 推荐、跟单、画像先做规则驱动，不做复杂机器学习。
5. 所有新增接口都要写 OpenAPI 注释和测试。
6. 不要破坏 V1 的 unified 层接口结构。
7. Any feature without audit logging is incomplete.
8. Every new feature must include at least one service-level test and one API-level test.
9. Keep provider logic separated from domain logic.
10. Keep AI logic separated from CRUD/business APIs.
11. Prefer simple explicit code over clever abstractions.

## 5. V2 仍然不做什么

V2 明确不做：
- 完整质检引擎
- 完整 ERP / OMS / WMS 深联
- 大规模自动群发
- 所有平台真实 API 全量接入
- 复杂多模型风控
- V3 管理分析与训练中心

## 6. V2 交付物

V2 完成后必须交付：
1. 客服工作台升级版
   - 推荐面板
   - 风险提示
   - 客户画像入口
2. 运营管理后台
   - 任务
   - 活动
   - 人群运营
3. 客户标签与画像系统
   - 规则打标
   - 人工修正
4. 推荐与跟单服务
   - 推荐结果
   - 跟单任务
5. 运营分析模块
   - 接待
   - 转化
   - 任务执行效果
6. 风险最小能力
   - 负面情绪
   - 高投诉倾向

## 7. 开发原则

- V2 仍然以“系统生成建议 + 人工确认执行”为主。
- V2 不建议大规模自动发。
- 所有新增模块必须能挂接到现有 unified domain model。
- 代码和目录设计必须为 V3 留接口，但不能提前实现 V3 业务逻辑。
