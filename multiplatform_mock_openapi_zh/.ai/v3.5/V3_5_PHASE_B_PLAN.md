## 三、真实数据刷新机制规划

### 3.1 为什么需要刷新机制

**Phase A 现状**：
- `refresh_from_provider` 需要手动调用
- 无刷新状态记录
- 无刷新失败处理
- 无刷新时间追踪

**Phase B 需求**：
- 建立可追踪、可恢复、可审计的刷新机制
- 避免数据无界增长
- 保持 snapshot 和 explain-status 语义稳定

### 3.2 手动 refresh 与受控定时 refresh 的边界

| 场景 | 推荐方式 | 说明 |
|------|----------|------|
| 开发/调试 | 手动 refresh | 开发者显式控制 |
| 测试环境 | 手动 refresh | 测试用例显式控制 |
| 生产环境初始化 | 手动 refresh | 运维显式控制 |
| 生产环境日常同步 | 受控定时 refresh | 按配置频率自动执行 |

**边界原则**：
- 定时刷新仅在生产环境启用
- 开发/测试环境必须保持手动控制
- 提供强制刷新接口供运维使用

### 3.3 刷新频率分级

| 数据类型 | 建议频率 | 理由 |
|----------|----------|------|
| inventory | 5-15 分钟 | 库存变化相对频繁 |
| order_audit | 10-30 分钟 | 订单审核状态变化中等 |
| order_exception | 30-60 分钟 | 异常状态变化较慢 |

**配置化原则**：
- 频率应通过配置文件控制
- 不同环境可配置不同频率
- 支持动态调整

### 3.4 刷新失败处理

| 失败类型 | 处理策略 |
|----------|----------|
| Odoo 连接失败 | 记录错误日志，保留上次 snapshot，不阻塞系统 |
| 认证失败 | 记录错误日志，发送告警，停止后续刷新 |
| 数据解析失败 | 记录错误日志，跳过问题记录，继续处理其他记录 |
| 数据库写入失败 | 记录错误日志，回滚当前批次，保留上次 snapshot |

**重试策略**：
- 不自动重试（避免雪崩）
- 由运维手动触发刷新或调整配置

### 3.5 避免重复写入和无界增长

| 问题 | 解决思路 |
|------|----------|
| 重复写入 | 基于 `sku_code + warehouse_code` 或 `order_id` 做去重判断 |
| 无界增长 | 设置 snapshot 保留策略（如保留最近 N 条或最近 N 天） |
| 历史数据清理 | 提供清理脚本，由运维定期执行 |

**注意**：具体实现方案需在编码阶段确认，不在规划阶段定死。

### 3.6 记录最近一次刷新状态

**建议记录字段**：

| 字段 | 说明 |
|------|------|
| `last_refresh_at` | 最近刷新时间 |
| `last_refresh_status` | 刷新状态（success/failed/partial） |
| `last_refresh_source` | 数据来源（odoo/mock） |
| `last_refresh_error_summary` | 错误摘要（如有） |
| `last_refresh_inventory_count` | 刷新的 inventory 数量 |
| `last_refresh_audit_count` | 刷新的 audit 数量 |
| `last_refresh_exception_count` | 刷新的 exception 数量 |

**存储位置**：
- 可新增 `IntegrationSyncStatus` 表
- 或使用配置存储（如 Redis）

### 3.7 保持 snapshot 和 explain-status 语义稳定

**原则**：
- 刷新机制不改变 snapshot 表结构
- 刷新机制不改变 explain-status 逻辑
- explain-status 继续基于 snapshot 工作
- 刷新失败时，explain-status 继续基于最近 snapshot 工作

## 四、Provider 选择机制规划

### 4.1 为什么需要 Provider 选择机制

**Phase A 现状**：
- Provider 通过显式注入传入 IntegrationService
- 无配置驱动切换能力
- 切换需要改代码

**Phase B 需求**：
- 支持通过配置切换 mock/real provider
- 支持环境隔离（开发/测试/生产）
- 保持 IntegrationService 简洁

### 4.2 切换逻辑应放在哪一层

**建议层级**：

```
配置层 (config) 
    ↓
工厂层 (provider_factory)
    ↓
Service 层 (integration_service)
```

**职责划分**：

| 层级 | 职责 |
|------|------|
| 配置层 | 定义 `ODOO_PROVIDER_MODE` 环境变量 |
| 工厂层 | 根据配置创建 mock 或 real provider |
| Service 层 | 接收 provider，不关心具体实现 |

### 4.3 哪些场景必须继续显式注入

| 场景 | 方式 | 理由 |
|------|------|------|
| 单元测试 | 显式注入 | 需要精确控制 mock 行为 |
| 集成测试 | 显式注入 | 需要精确控制测试数据 |
| 调试特定 provider | 显式注入 | 开发者需要精确控制 |

### 4.4 哪些场景可以允许受控配置切换

| 场景 | 方式 | 理由 |
|------|------|------|
| 生产环境 | 配置切换 | 运维通过配置控制 |
| 预生产环境 | 配置切换 | 运维通过配置控制 |
| 本地开发连接真实 Odoo | 配置切换 | 开发者通过配置控制 |

### 4.5 如何避免自动切换破坏测试和本地环境

**保护策略**：

| 策略 | 说明 |
|------|------|
| 默认值保护 | 默认使用 mock provider，避免误连生产 |
| 环境隔离 | 通过 `ODOO_PROVIDER_MODE` 明确区分环境 |
| 测试强制 mock | 测试代码强制使用 mock provider，不受配置影响 |
| 日志记录 | 切换时记录日志，便于排查 |

### 4.6 如何保持 integration_service 简洁

**设计原则**：
- IntegrationService 不直接读取配置
- IntegrationService 只接收 provider 接口
- 切换逻辑封装在工厂层
- Service 层代码不感知 mock/real 差异

## 五、order_exception 升级路线规划

### 5.1 当前为什么只能有限支持

**Phase A 实现方式**：
- 从 `sale.order` 模型读取订单数据
- 通过 `note` 字段关键词推断异常类型
- 无真实异常数据源

**问题**：
- Odoo 没有独立的"订单异常"模块
- 关键词推断不可靠
- 无法区分真实异常和普通订单

### 5.2 当前实现为什么不够稳定

| 问题 | 影响 |
|------|------|
| 关键词推断不可靠 | 可能误判或漏判异常 |
| 无真实异常字段 | 无法获取异常详情 |
| 所有订单都生成 exception | 数据冗余，失去异常意义 |
| 无法追踪异常处理状态 | 无法支持异常闭环处理 |

### 5.3 后续优先评估的真实来源

**候选来源优先级**：

| 优先级 | 来源 | 说明 |
|--------|------|------|
| P1 | `stock.picking` | 发货/配送记录，可推断延误异常 |
| P2 | `mail.activity` | 待办事项，可能包含异常标记 |
| P3 | `stock.return.picking` | 退货记录，逆向履约异常 |
| P4 | `crm.lead` | 客户投诉/问题记录 |
| P5 | `helpdesk.ticket` | 工单系统（如安装） |

**评估要点**：
- 数据是否真实存在
- 字段是否足够支撑异常语义
- 是否需要额外模块安装

### 5.4 如何定义"异常订单"的最小统一语义

**建议定义**：

| 字段 | 是否强依赖 | 说明 |
|------|------------|------|
| `order_id` | ✅ 强依赖 | 订单唯一标识 |
| `exception_type` | ✅ 强依赖 | 异常类型（delay/stockout/address/customs/other） |
| `exception_status` | ✅ 强依赖 | 异常状态（open/processing/resolved/cancelled） |
| `detected_at` | ✅ 强依赖 | 异常发现时间 |
| `source_model` | ⚠️ 辅助判断 | Odoo 来源模型 |
| `source_id` | ⚠️ 辅助判断 | Odoo 来源记录 ID |
| `detail_json` | ⚠️ 辅助判断 | 详细信息 |

### 5.5 在未拿到稳定来源前的口径克制

**API 口径**：
- 不宣称 `order_exception` 为稳定能力
- 返回数据时附带 `limited_support: true` 标识
- 文档明确说明当前为有限支持

**前端口径**：
- 不突出展示 exception 数据
- 标注"数据仅供参考"
- 不基于 exception 做自动化决策

## 六、运维与可观测性规划

### 6.1 refresh 成功/失败日志怎么记

**日志级别**：

| 事件 | 级别 | 内容 |
|------|------|------|
| 刷新开始 | INFO | 刷新类型、触发方式 |
| 刷新成功 | INFO | 刷新类型、记录数、耗时 |
| 刷新失败 | ERROR | 刷新类型、错误类型、错误摘要 |
| 部分失败 | WARN | 刷新类型、成功数、失败数、错误摘要 |

**日志格式建议**：
```
[INTEGRATION] refresh_started type=inventory trigger=manual
[INTEGRATION] refresh_success type=inventory count=38 duration_ms=1234
[INTEGRATION] refresh_failed type=inventory error=connection_timeout
```

### 6.2 如何记录 snapshot 数据来源

**建议字段**：

| 字段 | 存储位置 | 说明 |
|------|----------|------|
| `source_type` | snapshot 表 | 数据来源类型（odoo/mock/manual） |
| `source_sync_id` | snapshot 表 | 同步批次 ID |
| `source_raw_json` | snapshot 表 | 原始数据（已有 `source_json`/`detail_json`） |

### 6.3 如何记录最近同步时间

**存储方案**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 新增 `IntegrationSyncStatus` 表 | 结构化、可查询 | 需要新增表 |
| 使用 Redis | 快速、轻量 | 依赖外部服务 |
| 使用配置文件 | 简单 | 不适合高频更新 |

**建议**：新增 `IntegrationSyncStatus` 表，记录每次同步状态。

### 6.4 如何暴露最小运维信息

**最小运维信息范围**：

| 信息 | 暴露方式 | 说明 |
|------|----------|------|
| 最近同步时间 | 内部 API | `/api/integration/sync-status` |
| 同步状态 | 内部 API | success/failed/partial |
| 各类型记录数 | 内部 API | inventory/audit/exception count |
| 错误摘要 | 内部 API | 最近错误信息 |

**不暴露**：
- 详细错误堆栈
- 敏感配置信息
- 原始 Odoo 数据

### 6.5 管理侧最小只读信息

**建议展示**：

| 信息 | 展示位置 | 说明 |
|------|----------|------|
| 数据来源 | Admin Console | 显示当前使用 mock/real |
| 最近同步时间 | Admin Console | 显示各类型最近同步时间 |
| 同步状态 | Admin Console | 显示成功/失败状态 |
| 记录总数 | Admin Console | 显示各类型 snapshot 数量 |

### 6.6 后续问题排查支持

**排查能力**：

| 能力 | 实现方式 |
|------|----------|
| 查看同步历史 | 查询 `IntegrationSyncStatus` 表 |
| 查看错误日志 | 查询应用日志 |
| 查看原始数据 | 查询 snapshot 的 `source_json` |
| 手动触发刷新 | 调用内部 API |

## 八、Phase B 风险清单

| 风险 | 等级 | 说明 | 缓解措施 |
|------|------|------|----------|
| order_exception 真实来源不稳定 | ⚠️ 高 | 可能找不到合适的真实数据源 | 保持有限支持口径，明确标注 |
| 刷新机制可能过度复杂 | ⚠️ 中 | 定时、重试、增量等可能引入复杂度 | 保持最小实现，避免过度设计 |
| provider 切换可能影响测试/本地环境 | ⚠️ 中 | 配置错误可能误连生产 | 默认 mock，强制测试隔离 |
| 运维暴露粒度失控 | ℹ️ 低 | 可能暴露过多运维信息 | 明确最小范围，不做监控平台 |
| 数据增长与清理策略不明确 | ℹ️ 低 | snapshot 可能无限增长 | 规划阶段明确清理策略 |
