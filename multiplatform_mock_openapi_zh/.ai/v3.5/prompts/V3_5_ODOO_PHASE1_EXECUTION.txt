1. inventory snapshot
2. order audit snapshot

二、本轮必须坚持的架构

统一采用：

Odoo -> providers/odoo/real -> provider-sdk abstraction -> integration_service -> snapshot tables -> /api/integration/* -> frontend

不允许采用：

1. 前端直接请求 Odoo
2. API 直接透传 Odoo 原始返回
3. 跳过 snapshot 表直接返回 real provider 结果
4. 在 service 中写死 Odoo 原始字段逻辑而不经过 provider 抽象

五、建议的第一阶段真实接入对象

A. Inventory
从 Odoo 获取：
- sku_code
- warehouse_code
- available_qty
- reserved_qty
- status
- 原始 payload

B. Order Audit / Order State
从 Odoo 获取：
- order_id
- platform 或内部来源标识
- audit_status 或最接近的订单审核/状态字段
- audit_reason
- 原始 payload

C. Order Exception / Fulfillment Exception
从 Odoo 获取：
- order_id
- platform 或内部来源标识
- exception_type
- exception_status
- detail_json

注意：
如果 Odoo 本身没有与你当前命名完全一致的对象，不要硬找一模一样字段。
要做的是：
- 在 provider 层完成最小语义映射
- 再映射到 Omni-CSX 现有 snapshot 结构
